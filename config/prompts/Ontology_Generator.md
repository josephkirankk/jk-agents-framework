Your task is to create a **validated, production-ready ontology JSON** of **defects → root causes → corrective actions** for the machine specified below. The ontology must be comprehensive (≥90% real-world coverage), consistent, and directly usable for ML prediction, search, and operator guidance.

- **Machine**: {{MACHINE_NAME}}
- **Process context**: {{PROCESS_CONTEXT}} (e.g., cold pilgering of tubes, hot rolling line, CNC machining cell)
- **Operating environment**: {{ENVIRONMENT}} (e.g., shop floor with oil mist, ambient 10–45 °C, 24/7 operation)

## Coding rules (strict)

- **Defect codes**: `{{PREFIX}}.{{SUBSYS}}.{{COMPONENT}}.{{DEFECT}}`
  - Example: `HSP.GBX.GEARBOLT.LOOSE`
- **Root cause codes**: `RC.{{CATEGORY}}.{{CAUSE}}`
  - Example: `RC.LUBE.INSUFFICIENT`
- **Corrective action codes**: `CA.{{CATEGORY}}.{{ACTION}}`
  - Example: `CA.TORQUE.RETIGHTEN`
- All codes UPPERCASE, segments separated by `.`, segment words by `_` only, ASCII, max 60 chars.
- **Uniqueness required** across each list. No duplicate codes.
- **No placeholders** (e.g., “TBD”/“Autogenerated”).

## Content requirements

1. **Coverage goal (≥90%)**
   - For each relevant subsystem, include **the top recurring field defects** and **product/output defects** unique to this machine.
   - Typical target: **5–12 defects per subsystem** (more where necessary).
2. **Evidence-based**
   - Derive from **OEM manuals**, **reputable standards**, **vendor apps notes**, **peer-reviewed papers**, and **reliability handbooks**.
   - Prefer the last **10 years**; include earlier seminal sources if needed.
3. **Clarity & actionability**
   - **Defect text** describes the observable issue.
   - **Root cause text** is mechanistic/casual (why it happens).
   - **Corrective action text** is a clear, technician-executable step (what to do). Avoid vague language.
4. **Balance & realism**
   - Don’t overfit to a single vendor; capture common variants of the machine.
   - Include **installation/setup** and **changeover** errors where relevant (they’re frequent root causes).
   - Include **contamination**, **alignment**, **lubrication**, **electrical interconnect**, **controls/firmware**, and **network** faults.
5. **Product/quality defects (PROD)**
   - Include downstream product defects that originate in the machine (e.g., thickness variation, surface scoring, ovality), with realistic causes & actions.

## Validation rules (must perform before output)

- Ensure **every** `likely_root_causes[]` entry appears in `root_causes`.
- Ensure **every** `recommended_actions[]` entry appears in `corrective_actions`.
- Enforce **2–5** root causes and **2–5** actions per defect (unless you justify fewer in `meta.notes`).
- Guarantee **unique codes** and remove any duplicates.
- **No placeholders** (“TBD”, “autogenerated”, “lorem ipsum”).
- Update `meta.counts` accurately.

## Quality bar & writing style

- Write **concise, concrete, technician-friendly** sentences.
- Use consistent terminology across entries.
- Avoid manufacturer-specific jargon unless it’s a de-facto standard (then note synonyms).
- Prefer verbs in corrective actions: “Retighten…”, “Replace…”, “Align…”, “Flush…”.

## Citations

- Populate `meta.sources` with **8–15** high-quality sources (mix of OEM manuals, standards, textbooks, peer-reviewed articles, reputable vendors).
- Each item: `title`, `publisher`, `url`, `date_accessed`.
- If sources disagree, prefer consensus; note disagreements in `meta.notes`.

## Output formatting

- Return **only the JSON** conforming to the schema above.
- No markdown, no commentary.
- Use UTF-8, escaped as needed.
- Keep line length readable; indent with 2 spaces.
