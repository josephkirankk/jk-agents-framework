## Project Overview

**JK-Agents-Core** is a sophisticated multi-agent AI orchestration framework built on LangGraph and LangChain. It provides supervisor-based planning with specialized agent execution, multi-provider LLM support (Azure OpenAI, Google Gemini, Anthropic, OpenAI), and advanced memory management via ChromaDB.

## High-Level Architecture

### Multi-Agent Orchestration Model

The framework uses a **supervisor-worker pattern** where:

1. **Supervisor Agent** (`app/supervisor_builder.py`):
   - Receives user queries and creates execution plans
   - Breaks complex tasks into sequential/parallel steps
   - Assigns steps to specialized worker agents
   - Manages dependencies between steps

2. **Worker Agents** (`app/agent_builder.py`):
   - Specialized agents configured via YAML files
   - Each agent has specific prompts, models, and tools (MCP servers)
   - Uses ReAct (Reasoning + Acting) pattern with tool calling
   - Can be normal conversational agents or tool-enabled agents

3. **Plan Executor** (`app/planner_executor.py`):
   - Executes plans generated by supervisor
   - Handles timeouts, retries, and error recovery
   - Manages step dependencies and parallel execution

### Memory Architecture

The system has **two parallel memory systems**:

**1. Conversation Memory** (`app/conversation_tracker.py`, `app/simple_conversation_memory_fixed.py`):
- Turn-based conversation history stored in ChromaDB
- Automatically injected into agent prompts as context
- Includes summarization capabilities when context grows large
- Format: `[Turn-1] User: ... [Turn-1] Agent: ...`

**2. LangGraph Checkpointing** (`app/checkpointer_manager.py`):
- State persistence for agent execution graphs
- Thread-based state management (`thread_id` parameter)
- Enables multi-turn conversations with stateful agents
- Backed by ChromaDB or in-memory storage

These systems work together: conversation memory provides historical context while checkpointing maintains execution state.

### Multi-Provider Model System

**Model ID Format** (`app/config_model_format.py`, `app/agent_builder.py`):
```
Format: "provider:model_name" or "provider/model_name" (for LiteLLM)
Examples:
  - azure_openai:gpt-4.1
  - google:gemini-2.0-flash-exp
  - anthropic:claude-sonnet-4
  - openai:gpt-4o
  - gemini/gemini-1.5-pro  (LiteLLM format)
```

**Provider Resolution Logic**:
1. **Enhanced LiteLLM Wrapper** (`app/enhanced_litellm_wrapper.py`):
   - Handles all LiteLLM-format models (`provider/model`)
   - Supports multimodal inputs (images, files)
   - Primary choice for Google Gemini models

2. **Azure LiteLLM Wrapper** (`app/azure_litellm_wrapper.py`):
   - Specialized for Azure OpenAI deployments
   - Used when model starts with `azure/`

3. **Native LangChain Providers**:
   - Direct integration for OpenAI, Anthropic
   - Fallback for standard models

### Configuration System

**YAML-Based Configuration** (`config/*.yaml`):
```yaml
models:
  default: "azure_openai:gpt-4.1"      # Default model for agents
  supervisor: "azure_openai:gpt-4.1"    # Model for supervisor
  temperature: 0.1                       # Global temperature

business_context: |
  Shared context injected into all agent prompts

supervisor:
  name: "supervisor"
  model: "azure_openai:gpt-4.1"
  prompt: |
    Supervisor prompt with {{placeholders}}

agents:
  - name: "worker_name"
    description: "Agent description for supervisor"
    model: "google:gemini-2.0-flash-lite-001"
    agent_type: "react"  # or "normal"
    prompt: |
      Agent-specific prompt
    mcp_servers:
      tool_name:
        transport: "stdio"
        command: "python"
        args: ["-m", "app.mcp_tool"]
```

**Key Configuration Files**:
- `config/python_exec_agent_working.yaml`: Production-ready Python execution
- `config/multi_provider_agent.yaml`: Multi-provider demonstration
- `config/visiting_card_extractor.yaml`: OCR and vision processing
- `config/gemini-test.yaml`: Google Gemini integration testing

### MCP (Model Context Protocol) Integration

**MCP Servers** (`app/mcp_loader.py`):
- Extensible tool system for agents
- Supports stdio, HTTP, and SSE transports
- Tools are loaded per-agent based on YAML configuration

**Built-in MCP Servers**:
- `python_runner`: Python code execution via Deno
- `conversation_manager`: Conversation cleanup/summarization
- Custom tools in `tools/` directory

**Tool Loading** (`app/python_tool_loader.py`):
- Python functions can be exposed as tools
- Automatic schema generation from type hints
- Validation and error handling

### File Storage and Reference System

**File Management** (`app/file_storage_manager.py`):
- Files uploaded via API are stored temporarily with reference IDs
- Reference IDs injected into prompts for agent access
- Supports images, CSV, text files, and documents
- Automatic cleanup after processing
- Used for multimodal agent interactions

**Image Compression** (`app/image_compression.py`):
- Automatic image resizing for API limits
- Maintains aspect ratio and quality
- Reduces token usage for vision models

## Critical Implementation Details

### Agent Type Selection

When configuring agents in YAML:
- **`agent_type: "react"`**: Use for agents that need tool calling (most agents)
  - Enables ReAct pattern (Reasoning + Acting)
  - Required for MCP tools, Python execution, etc.
  - Built using `create_react_agent()` from LangGraph

- **`agent_type: "normal"`**: Use for pure conversational agents
  - No tool calling capabilities
  - Used for final response formatting, text generation
  - Simpler execution path

### Google Gemini Specific Handling

**Tool Compatibility** (`app/gemini_schema_filter.py`):
- Gemini has strict tool schema requirements
- Framework automatically filters incompatible tool parameters
- Removes recursive schemas, complex nested types
- Falls back to non-tool binding if tools are incompatible

**Model Format Conversion** (`app/config_model_format.py`):
- Accepts both `google:gemini-*` and `gemini/gemini-*` formats
- Automatically converts to LiteLLM format internally
- Handles model name variations (gemini-2.0-flash-exp, etc.)

### Memory and Context Flow

**Context Injection Pattern**:
1. User sends query to `/query` endpoint
2. System checks for existing `thread_id` or generates new one
3. Conversation memory retrieves previous turns
4. Memory context injected into prompts as `{{conversation_context_metadata}}`
5. Supervisor creates plan based on query + memory
6. Workers execute with injected context from their dependencies
7. Response stored in conversation memory with turn tracking

**Placeholder System** (`app/template_utils.py`):
- Prompts support Jinja2-style placeholders: `{{variable_name}}`
- Built-in placeholders: `{{business_context}}`, `{{conversation_context_metadata}}`, `{{agents}}`, `{{mcpservers}}`, `{{dependent_request_responses}}`
- Custom placeholders can be added via configuration

## Common Development Tasks

### Adding a New Agent

1. Create or modify YAML config in `config/`:
```yaml
agents:
  - name: "new_agent"
    description: "What this agent does"
    model: "azure_openai:gpt-4.1"
    agent_type: "react"
    prompt: |
      Your specialized prompt here
    mcp_servers:
      tool_name:
        transport: "stdio"
        command: "python"
        args: ["-m", "module.name"]
```

2. Test the agent:
```bash
curl -X POST "http://localhost:8000/worker" \
  -H "Content-Type: application/json" \
  -d '{"agent_name": "new_agent", "input": "test input", "config_path": "config/your_config.yaml"}'
```

### Adding a New MCP Tool

1. Create Python module in `app/` or `tools/`
2. Implement MCP protocol (stdio, HTTP, or SSE)
3. Add to agent configuration:
```yaml
mcp_servers:
  your_tool:
    description: "Tool description"
    transport: "stdio"
    command: "python"
    args: ["-m", "app.your_tool"]
```

### Switching LLM Providers

Update `.env` with new provider credentials and modify `models` section in YAML:
```yaml
models:
  default: "google:gemini-2.0-flash-exp"  # Change provider
  supervisor: "anthropic:claude-sonnet-4"
```

### Debugging Memory Issues

1. Enable memory logging in `.env`:
```bash
MEMORY_LOGGING_ENABLED=true
MEMORY_LOGGING_INCLUDE_CONTENT=true
```

2. Check memory stats:
```bash
curl http://localhost:8000/memory/stats
```

3. Review logs in `memory_logs/` directory

4. Clear specific thread if needed:
```bash
curl -X POST "http://localhost:8000/memory/clear/thread123"
```

## Architecture Diagrams (Text)

```
Request Flow:
Client Request → FastAPI (/query) → load_app_config() → build_agents_map()
    ↓
Supervisor Agent (plan creation) → JSON plan with steps
    ↓
Plan Executor → Worker Agent 1 → Worker Agent 2 → ... → Worker Agent N
    ↓
Response aggregation → Client Response

Memory Flow:
User Query → get_or_create_thread_id() → retrieve conversation context from ChromaDB
    ↓
inject context into prompt → Agent Execution → store response in ChromaDB
    ↓
Next turn retrieves enhanced context

Model Resolution:
model_id string → parse_model_id() → check format (colon vs slash)
    ↓
is_litellm_model()? → Yes → EnhancedLiteLLMChat
    ↓                → No → Native LangChain provider
    ↓
Model instance → Agent builder → ReAct/Normal agent
```
