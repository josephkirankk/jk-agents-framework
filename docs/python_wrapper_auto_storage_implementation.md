# Python Wrapper with Auto-Storage Implementation

## Summary

Successfully implemented a Python wrapper MCP server that automatically stores large datasets generated by Python code execution, eliminating the need for manual storage calls and preventing context overflow.

## Problem Statement

The original system had a critical limitation:
- The Deno-based Python runner (`jsr:@pydantic/mcp-run-python`) truncates large output to prevent context overflow
- When generating 5000 records, only 2-5 records were being returned to the LLM
- The agent would then manually call `store_large_dataset` with the truncated data
- Result: Only 2-5 records stored instead of 5000

## Solution

Created a Python wrapper MCP server (`app/mcp_python_wrapper.py`) that:
1. Executes Python code directly (no Deno dependency)
2. Automatically detects large datasets (>5 items)
3. Stores them in the database using `LargeDataStorage`
4. Returns only a preview + reference ID to the LLM

## Implementation Details

### 1. Python Wrapper Server (`app/mcp_python_wrapper.py`)

**Key Features:**
- Direct Python execution using `exec()` and `eval()`
- Automatic dataset detection for lists and dicts with >5 items
- Integration with existing `LargeDataStorage` system
- Preview generation with reference IDs

**Code Execution Strategy:**
```python
# Try eval first (for expressions)
try:
    code_obj = compile(python_code, '<string>', 'eval')
    dataset = eval(code_obj, restricted_globals, local_vars)
except SyntaxError:
    # Fall back to exec (for statements)
    exec(python_code, restricted_globals, local_vars)
    # Extract result from common variable names
    dataset = local_vars.get('orders') or local_vars.get('data') or ...
```

**Auto-Storage Logic:**
```python
if isinstance(dataset, list) and len(dataset) > 5:
    reference_id = f"ref_{uuid.uuid4().hex[:12]}"
    storage.store_large_data(
        reference_id=reference_id,
        tool_name="run_python_code",
        data=dataset,
        metadata={...}
    )
    return preview_response(dataset, reference_id)
```

### 2. Configuration Update (`config/large_data_mcp_demo.yaml`)

**Before:**
```yaml
python_runner:
  description: "Python code execution for generating datasets"
  transport: "stdio"
  command: "deno"
  args:
    - "run"
    - "-N"
    - "jsr:@pydantic/mcp-run-python"
```

**After:**
```yaml
python_runner:
  description: "Python code execution with auto-storage for large datasets"
  transport: "stdio"
  command: "python"
  args:
    - "-m"
    - "app.mcp_python_wrapper"
```

### 3. Agent Prompt Update

**Updated to reflect automatic storage:**
```
AUTOMATIC STORAGE WORKFLOW:
1. Use run_python_code to generate the COMPLETE dataset
2. Return the full dataset as a Python list or dict
3. The system AUTOMATICALLY stores datasets with >5 items
4. You receive back: {reference_id, preview, total_count, description}
5. Share the reference_id and preview with the user

IMPORTANT:
- Just return the Python list/dict from your code
- DO NOT call store_large_dataset manually - it happens automatically
- The system detects large datasets and stores them for you
```

## Benefits

1. **No Context Overflow:** Large datasets never bloat the LLM context
2. **Automatic Storage:** No manual `store_large_dataset` calls needed
3. **Consistent Behavior:** Works for any dataset size
4. **Better Performance:** Direct Python execution (no Deno overhead)
5. **Simplified Agent Logic:** Agent just generates data, storage is automatic

## Architecture

```
[Agent generates Python code]
        ↓
[Python Wrapper MCP Server]
        ↓
[Execute Python code directly]
        ↓
[Detect large dataset (>5 items)]
        ↓
[Auto-store in LargeDataStorage]
        ↓
[Return preview + reference_id to LLM]
        ↓
[Agent shares reference_id with user]
```
